## 3. ローカル開発環境

ブラウザのGitHub Web編集だけでも運用できますが、表示確認やデザイン変更時はローカルが便利です。

```bash
# 前提: Node.js 22 LTS（バージョン管理は Volta か fnm 推奨）
npm ci          # 依存をインストール（package-lock.json に忠実）
npm run dev     # 開発サーバ（http://localhost:4321 でプレビュー）
npm run build   # 本番ビルド（出力は dist/）
npm run preview # build した dist/ をローカル配信して最終確認
```

> ⚠️ **Windowsローカルのビルドは万能ではない。** Windowsはファイル名の大文字小文字を区別しないため、
> 画像パスの大小ズレ（`.JPG` 問題）を**ローカルでは見逃す**。最終確認は必ずCI（Ubuntu）でも行うこと（§7参照）。

## 4. デプロイのしくみ

[.github/workflows/deploy.yml](../.github/workflows/deploy.yml) が以下を行う。

- **build ジョブ**: `master`/`feat/astro-migration` への push・全PR・手動実行で常に走る。Ubuntu上で `astro build`。
- **deploy ジョブ**: `master` のときだけ走り、`dist/` を GitHub Pages へ公開する。

ポイント:
- **build が成功したときだけ deploy する**構造。書式ミスでビルドが落ちれば公開されず、本番は直前の状態を維持（安全装置）。
- 公開は GitHub Pages の **Source = "GitHub Actions"** 設定が前提（リポジトリ Settings > Pages）。
- **ロールバック**: 問題が出たら `git revert <コミット>` で戻して push すれば再デプロイされる。

### 運用フロー
通常は **master編集 → 自動build&deploy → 数分で公開**。大きな変更はブランチ＋PRでCIの緑を確認してからマージするのが安全。

## 5. コンテンツコレクション（スキーマ）

定義は [src/content/config.ts](../src/content/config.ts)。

### `members`（部員プロフィール）
ファイル `src/content/members/member_<id>.md`。**ファイル名の `<id>` がURLセグメント**（例 `member_bansho` → `/gallery/member_bansho/`）。

| フィールド | 型 / 既定 | 説明 |
|---|---|---|
| `nickname` | string（必須） | 表示名 |
| `icon` | string（必須） | アイコン画像パス |
| `iconWidth` | number = 130 | アイコン幅 |
| `bio` | string（必須） | 自己紹介。`<br>` 等のHTML可 |
| `order` | number = 99 | （予備）並び順 |
| `managed` | boolean = **true** | **true=作品棚を自動生成 / false=自動生成しない（独自ページを使う）** |

### `works`（個別作品）
ファイル `src/content/works/<member_id>/<slug>.md` → `/gallery/<member_id>/<slug>/`。

| フィールド | 型 / 既定 | 説明 |
|---|---|---|
| `title` | string（必須） | 作品名（見出し＆`<title>`） |
| `order` | number（**必須**） | 作品棚の並び順 |
| `card.thumb` | string（必須） | サムネ画像パス |
| `card.width` | number = 300 | サムネ幅 |
| `images[]` | `{src, width?}`（既定[]） | 作品ページに並べる画像 |
| `heading` | string = "どういう作品？" | 画像下の見出し。空文字で非表示 |
| `cardTitle` | string（任意） | カード用の別見出し |
| `related` | `{href,label,thumb,width=150}`（任意） | 関連リンク（学祭ページ等） |
| `external` | string（任意） | 指定すると**個別ページを作らず**カードを外部URLへリンク |

## 6. 部員ページの3パターン（重要）

部員棚のレンダリング方法は3つあり、`managed` フラグと配置で決まる。

| パターン | 条件 | 作品棚の作り方 | 例 |
|---|---|---|---|
| **A. 自動生成** | `managed: true` | `works/<id>/*.md` を置くだけ。カード3列グリッド＋個別作品ページを [gallery/[member]/index.astro](../src/pages/gallery/[member]/index.astro) が自動生成 | 番匠, ねぎ, ZEKE, レーロー, 石政 |
| **B. 独自レイアウト** | `managed: false` | 自動生成から除外。手書きの `src/pages/gallery/member_<id>/index.astro` を用意（個別作品ページは works から自動生成されるものを使うことも可） | CUDA狐, まめ大福, もっちー, くるぽんきゅー |
| **C. 完全手書き / passthrough** | コレクション未登録、または `public/` 配下 | `.astro` 直書き、もしくは自己完結HTMLを `public/` に置いてそのまま配信 | anthony（手書きインライン）, ヤスタジャン（public/passthrough） |

> パターンBで `managed: true` のまま手書き `index.astro` を置くと**ルート衝突でビルドエラー**になる。独自ページを作るときは必ず `managed: false` にすること。

## 7. タスク別の手順

### 7-1. 新しい部員の作品棚を追加する（標準＝パターンA）
3か所を編集する：

1. **プロフィール**: `src/content/members/member_<id>.md` を作成（§5のスキーマ）。`managed: true`。
2. **作品**: `src/content/works/<id>/*.md` を作成（最低1枚）。画像は `public/images/gallery/member_<id>/` へ。
3. **作品集トップの一覧に追記**: ⚠️ ここが自動化されていない。
   [src/pages/gallery/index.astro](../src/pages/gallery/index.astro) の `members` 配列に、アイコン・名前・bio・リンク（`/gallery/member_<id>/`）のエントリを1つ追加する。
4. アイコン画像を `public/images/gallery/thumbnail/<id>_icon.png` に置く。

### 7-2. 独自レイアウトの部員（パターンB）
- プロフィールmdを `managed: false` で作る。
- `src/pages/gallery/member_<id>/index.astro` を手書き（既存の [member_anthony/index.astro](../src/pages/gallery/member_anthony/index.astro) や member_cudafox をコピーして改造）。
- 作品集トップへの追記（7-1の手順3・4）は同様に必要。

### 7-3. トップの「最新情報」を更新
[src/pages/index.astro](../src/pages/index.astro) の `news` 配列の先頭に
`{ date, text, link, linkText }` を1件追加（直書き）。「代表作」も同ファイル内。

### 7-4. ワークショップ／コンテスト／学祭の記事を新規作成
コレクションではなく**個別の `.astro` ページ**。
既存の近い記事（例 [src/pages/about/workshop_26_kagakumiraikan.astro](../src/pages/about/workshop_26_kagakumiraikan.astro)）を
コピーして新ファイルを作り、内容を書き換える。一覧ページ（[about/index.astro](../src/pages/about/index.astro) 等）からのリンクも忘れずに追加する。

### 7-5. 自由制作（passthrough）ページ
共通テンプレートに縛られない自己完結ページは `public/gallery/<…>/` にHTML/CSS/JSをそのまま置く。
`public/` はバイト単位でそのまま配信されるため、URLも完全一致で保てる。
作品集トップにカードを出す場合は 7-1 手順3 と同じく一覧配列へ追記する。

## 8. SEO / メタ情報

各ページは [BaseLayout.astro](../src/layouts/BaseLayout.astro) を使い、props で制御する。

```astro
<BaseLayout
  title="ページタイトル｜大阪大学レゴ部"   {/* <title>・og:title */}
  description="120字程度の説明"            {/* meta description・og:description */}
  ogImage="/images/top/xxx.jpg"          {/* OGP画像。未指定はロゴ */}
  bodyClass="left-sidebar"               {/* homepage / left-sidebar / no-sidebar */}
>
```

- `canonical`・OGP・Twitter Card・`lang="ja"` は BaseLayout が自動付与。
- `sitemap.xml` は `@astrojs/sitemap` がビルド時に自動生成（`/sitemap-index.xml`）。
- GA4タグは BaseLayout に直書き。測定IDを変える場合はここを編集。

## 9. 既知の落とし穴・注意

- ⚠️ **画像の大文字拡張子（`.JPG` 等）**: Linux本番は大文字小文字を区別する。参照パスは実ファイル名と**完全一致**させること。Windowsローカルでは気づけないので、CI（Ubuntu）のビルド結果で確認する。
- **内部リンクはクリーンURL（末尾スラッシュ）**で書く（例 `/gallery/member_bansho/`）。`.html` は付けない。
- **frontmatterのインデント崩れ／全角スペース**はビルドエラーの定番。
- **巨大リポジトリ**（画像が多い）。Astroの画像最適化は使わず `public/` で無加工配信しているため、ビルドは数分かかることがある。

### トラブルシュート
- **ビルドが赤（失敗）** → GitHub の Actions タブで該当runのログを確認。多くは frontmatter書式 or 画像パスの大小ズレ。直して再push。
- **公開されない** → Actionsが緑か、Pages の Source が "GitHub Actions" かを確認。
- **全体を戻したい** → `git revert <コミット>` で前進的にロールバック。
